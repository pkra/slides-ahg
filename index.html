Authoring with accessible web documents in mind.

The case for web documents.

  • Most accessible.
  • Most flexible.
  • Most sustainable.
  • HTML: Accessible by default; you break it, nobody else
  • Browers: a powerful & universal platform
  • Robust: good web documents age well, e.g., the first page

Do one thing and do it well

  • Word etc.
  • LaTeX etc.
  • Markdown etc.

It doesn't matter which as long as you do a good job with it.

  • MS Word: Microsoft tips WebAIM tips
  • LaTeX: as few packages as possible, no hacks etc.

Keep it simple, stupid.

  • Native content -- stay in your tool
  • Simple layout -- do less, get more
  • Convert early, convert often
  • e.g., MS Word: prefer Word's built-in Math authoring rather than external tools.
  • e.g., LaTeX: don't use tabular environments or add manual breaks just to manipulate minutiae of a PDF page, don't use "fancy" packages like fancyhdr
  • switch things around: take intervals where you only work with HTML output, don't even look at print output

Get from here to there

  1. Word => pandoc
  2. LaTeX => pandoc, latexml (also: tex4ht, plastex, tralics)
  3. Markdown => dozens, e.g., pandoc, Writage (MS Word plugin)
  • Libre Office: (don't) try the HTML export, e.g., with LibreOffice Math Guide -- tag soup trying to preserve all layout, inlined images etc
  • LaTeX (others)
    • tex4ht tex4ht tutorial
    • For PDF (untested): accsup, pdfx
